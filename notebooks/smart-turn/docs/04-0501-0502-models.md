# 04 — `smart-turn-zh` 0501 → 0502

**Status:** 0501 and 0502 trained and compared; AP did **not** strictly
increase. This doc records what we shipped, why the comparison is
ambiguous, and the queued next steps.

## Datasets

| Snapshot | Train | Val | Test | Total | Source |
|---|---|---|---|---|---|
| `smart-turn-zh-0501` | 1256 | 155 | 158 | **1569** | wavekat-platform human labels only (zh) |
| `smart-turn-zh-0502` | 1470 | 185 | 181 | **1836** | 0501 + ~267 RAMC uncertainty-mined clips, hand-confirmed on platform |

The 0502 increment came from
[`notebooks/smart-turn-mining/`](../../smart-turn-mining/) Tier 1
mining + the 06b uncertainty-queue upload path: the current zh
checkpoint scored RAMC candidates, the 300 clips closest to the
operating threshold went to the platform as `pre_labeled`, were
human-confirmed (or flipped), then re-exported via
`wk exports adapt smart-turn` into `datasets/smart-turn-zh-0502/`.

## Models trained from each snapshot

Two variants per snapshot — same recipe as
`02_a_train_baseline.ipynb` / `02_b_train_specaugment.ipynb`, only the
`EXPORT_DIR` differs. Checkpoints under
`checkpoints/smart-turn-zh-0501/{baseline,specaugment}/` and
`checkpoints/smart-turn-zh-0502/…`. Each writes its own
`threshold.json`; `03_compare.ipynb` reads both pairs in one pass.

## Test PR — what we saw

Numbers below are AP read off `03_compare.ipynb`'s overlaid PR curves.
Each model is evaluated on **the test split that ships with its own
snapshot**, which is the trap — see "Why this isn't apples-to-apples"
below.

| Model | 0501 test AP | 0502 test AP | Δ |
|---|---|---|---|
| `wavekat-baseline` | 0.959 | 0.935 | **−0.024** |
| `wavekat-specaugment` | **0.990** | 0.909 | **−0.081** |
| `pipecat-v3` (zero-shot) | 0.961 | 0.931 | −0.030 |

So the headline is: more data + the same recipe produced **lower** AP
on every model, including the frozen pipecat-v3 baseline that we
didn't touch.

## Why this isn't apples-to-apples

The pipecat-v3 numbers are the proof: that model is a fixed ONNX
artifact — its weights did not change between 0501 and 0502, but its
test AP dropped 3pt anyway. **The 0502 test split is harder than the
0501 test split**, not because we labeled badly, but because of how it
was constructed:

- 0502 added **23 new test clips** (158 → 181). Those clips were drawn
  from the 300 RAMC near-threshold uncertainties — by construction the
  hardest, most ambiguous audio in the mining pool. Stratified random
  sampling (the default `wk exports create … --split random`) routed
  ~10% of them into test.
- The mined RAMC distribution (spontaneous F2F Mandarin, real
  disfluencies, real overlap) is also different from the wavekat
  product-audio distribution that dominated the 0501 test set.

Net effect: the 0502 test split is closer to the deployment
distribution we actually care about, but it isn't comparable to the
0501 test split. Any AP delta reads as both "model changed" and "test
changed" stacked together.

## What we don't know yet (and need to)

Before chasing the regression, disentangle the two effects:

1. **Score the 0501 specaugment checkpoint on the 0502 test split.**
   If its AP also drops to ~0.91, the regression is test-shift, not
   model-shift, and 0502 is fine (or even better — it learned a harder
   distribution and held steady).
2. **Score the 0502 specaugment checkpoint on the 0501 test split.**
   If its AP matches or beats the 0501 model on its own test, the new
   data didn't hurt the easy distribution, only the hard one.
3. **Per-source AP on the 0502 test split** — split test rows by
   `source_file_sha256` provenance (wavekat product audio vs. mined
   RAMC) and report AP per bucket. This tells us exactly where
   specaugment is leaking precision/recall.

Concrete action: add an "eval-on-other-snapshot" cell to
`03_compare.ipynb` (or a sibling `03_b_cross_eval.ipynb`) that loads
both snapshots' test parquets and scores every checkpoint on every
test set. Until that table exists, the AP regression is
uninterpretable.

## Next steps for "more high-quality data"

The plan in [`02-more-data.md`](02-more-data.md) is still the right
plan; 0502 just exposed two operational gaps in how we run it.

### Gap 1 — Freeze a real-distribution test set, then never touch it

The biggest mistake in the 0501 → 0502 cycle was letting the export
pipeline route new data into the test split. That makes the v1 ship
bar (test F1 + bootstrap CI from `MISSION.md`) impossible to track
across versions.

**Fix:** define `smart-turn-zh-test-frozen` as its own export, drawn
once from a stratified mix of (a) wavekat product audio and (b) mined
real-conversation audio (RAMC + later AISHELL-4), audit by ear, then
freeze. From 0503 onward, training snapshots only grow train/val;
**test is loaded from the frozen export, separately**. Update
`01_load_export.ipynb` to support a second `TEST_DIR` override.

Target frozen-test size: ≥ 500 clips with the source-bucket split
recorded so we always report per-bucket AP alongside aggregate AP.

### Gap 2 — Diversify mining sources before mining harder

Uncertainty mining (06b) is high-leverage per clip but
self-reinforcing: the model picks what it's already weak on, so the
distribution drifts toward whatever single corpus we mine from. 0502
made the test set 13% RAMC-near-threshold, but the train set is still
overwhelmingly wavekat product audio.

In rough priority for the next cycle (0503):

1. **Two-models-agree pool from RAMC.** `06_upload_to_platform.ipynb`
   already exists for this. Bigger volume, easier review (auto-labels
   are usually right), broadens the train distribution without picking
   only the hardest cases. Target: +1–2K clips into train.
2. **Continue uncertainty mining**, but stratified by session and
   speaker so a single chatty session can't dominate. Target:
   +300–500 clips, balanced.
3. **Bring in AISHELL-4** (per `02-more-data.md` Tier 1). Meeting
   audio with overlap stresses the model differently from RAMC's
   call-style F2F. Same `01_parse_*.ipynb` shape as the RAMC
   pipeline.
4. **Tier 3 — real wavekat product audio.** Slowest per-sample,
   highest distribution-match. Even 200 clips/week compounds; this is
   the only source allowed to grow the frozen test set.

### Gap 3 — Stop blindly trusting AP as "the metric"

`MISSION.md` ships on F1 (with bootstrap 95% CI) and continuation-class
recall, not AP. AP is a useful curve summary but it weights operating
points we'd never deploy at. Two concrete changes for 0503:

- `03_compare.ipynb` should print **F1 ± bootstrap CI** as the headline
  number for every checkpoint, with AP as a secondary diagnostic.
- For each shipped model, also print **continuation-class recall** at
  the chosen threshold — the failure mode users hate most per
  `MISSION.md`.

The exit criterion stays what `MISSION.md` says: test F1 ≥ 0.80 with a
bootstrap-95% CI lower bound ≥ 0.75 on a ≥ 500-clip real-human test
set, strictly beating `pipecat-v3` zero-shot and `specaugment` with
non-overlapping CIs. 0502's AP on its own test (0.91) is *consistent*
with that bar but cannot prove it — the test set is too small (181)
and too entangled with the train mining run.
